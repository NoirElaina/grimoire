# 产品规范提示词

让 AI 做产品时，最常见的问题是输入太模糊。

给 AI 一句"帮我做一个在线商城，要好看，要完善，要能用"，它会自由发挥：自己选框架、自己定功能范围、自己猜数据模型、自己设计接口。结果看起来什么都有，但每个部分都差一点——技术栈和你的团队不匹配、功能做了一堆不需要的、核心流程的边界没考虑。

更稳的做法是把提示词写成一份工程输入：产品要解决什么问题、给谁用、哪些功能必须做、哪些明确不做、用什么技术栈、代码怎么组织、数据怎么存、接口怎么设计、怎么判断完成。

## 模糊输入 vs 工程输入

模糊输入：

```text
帮我做一个在线商城，要好看，要完善，要能用。
```

AI 看到这句，只能猜。它不知道你要 B 端还是 C 端，不知道技术栈，不知道哪些功能优先，不知道什么算"完善"。

工程输入：

```text
我要做一个面向 C 端用户的秒杀商城 MVP。

前端使用 Vue 3 + Vite + TypeScript + Pinia + Element Plus。
后端使用 Spring Boot 3 + MyBatis-Plus + MySQL 8 + Redis + RabbitMQ + JWT。

必须实现：登录、商品列表、商品详情、下单、库存扣减、订单超时取消。
暂不实现：在线支付、优惠券、物流、多商户、微服务拆分。

先输出 PRD、数据模型、接口设计、任务拆分和验收标准，不要直接写代码。
```

第一种让 AI 自由发挥。第二种让 AI 在你的边界里设计。

## 一份完整的提示词长什么样

不要一节一节讲理论。直接看一个真实例子，然后拆解每个部分为什么这么写。

```text
你是一名资深产品经理、Java 后端架构师、Vue 前端架构师和工程负责人。

我要做一个秒杀商城 MVP。
请先不要写代码，先输出产品规范和技术实现方案。

## 产品背景

我要做一个用于学习 Java 后端工程能力的秒杀商城。
它不是商业级大平台，但要覆盖真实后端项目里的核心工程问题：
登录、商品、库存、订单、缓存、MQ、事务、幂等、异常和接口设计。

## 目标用户

- 普通用户：浏览商品、查看详情、下单。
- 管理员：维护商品和库存。

## 必须实现

- 用户注册和登录。
- JWT 鉴权。
- 商品列表。
- 商品详情。
- 创建订单。
- 库存扣减。
- 订单超时取消。
- RabbitMQ 发送订单创建事件。
- RabbitMQ 延迟消息处理超时订单。
- Redis 缓存商品详情。
- MySQL 保存用户、商品、库存、订单。

## 暂不实现

- 在线支付。
- 优惠券。
- 物流。
- 多商户。
- 微服务拆分。
- 分布式事务。

## 技术栈

前端：
- Vue 3 + Vite + TypeScript + Pinia + Vue Router + Element Plus + Axios

后端：
- Spring Boot 3 + Java 17 + MyBatis-Plus + MySQL 8 + Redis + RabbitMQ + JWT

架构约束：
- 单体分层架构。
- 不使用 JPA。
- 不使用微服务。
- 不使用前端 SSR。

## 技术要求

- 后端按 controller、service、mapper、entity、dto、config、common 分层。
- 前端按 views、components、api、stores、router、types 分层。
- 所有写操作必须说明事务边界。
- 创建订单必须考虑库存不足、重复提交、事务回滚。
- MQ 消息必须说明 exchange、queue、routing key、消息体。
- 消费者必须说明幂等策略。
- Redis 必须说明 key 命名、TTL、缓存未命中和缓存更新。
- 接口必须给出请求参数和响应格式。
- 数据库表必须给出字段、类型、索引和说明。

## 输出格式

1. 产品一句话说明
2. 用户角色
3. 核心业务流程（文本步骤）
4. 功能模块
5. 页面清单
6. 后端接口清单
7. 数据库表设计
8. Redis 缓存设计
9. RabbitMQ 消息设计
10. 事务设计
11. 幂等设计
12. 异常场景
13. 前端目录结构
14. 后端目录结构
15. 开发阶段拆分
16. 每个阶段的验证方式
17. 端到端验收用例
18. 风险点
19. 需要我确认的问题

## 输出约束

- 不要写空泛描述。
- 不要直接写代码。
- 不要引入未声明技术。
- 不要扩大范围。
- 如果缺信息，最多问 5 个问题。
- 如果可以合理假设，请写在"假设"里。
- 每个验收标准必须能被测试。
```

## 拆解：每个部分为什么这么写

### 角色设定

```text
你是一名资深产品经理、Java 后端架构师、Vue 前端架构师和工程负责人。
```

不是让 AI"扮演专家"走形式。而是告诉它：这个项目同时涉及产品设计、后端架构、前端架构和工程管理，输出时要覆盖所有这些视角。

如果只写"你是一名程序员"，AI 会只输出代码思路，不会输出产品流程和验收标准。

### 产品背景

```text
我要做一个用于学习 Java 后端工程能力的秒杀商城。
它不是商业级大平台，但要覆盖真实后端项目里的核心工程问题：
登录、商品、库存、订单、缓存、MQ、事务、幂等、异常和接口设计。
```

这段的作用是划定范围。"学习用"说明不需要生产级高可用，"覆盖核心工程问题"说明不能只做个空壳。AI 读完就知道该往什么深度做。

如果只写"做一个商城"，AI 可能给你加支付、物流、客服、推荐系统——因为它觉得"商城应该有这些"。

### 必须实现 + 暂不实现

```text
必须实现：
- 用户注册和登录。
- JWT 鉴权。
- 商品列表。
- ...

暂不实现：
- 在线支付。
- 优惠券。
- 物流。
- ...
```

这两段同样重要。

"必须实现"让 AI 知道功能范围。"暂不实现"让 AI 知道边界。不写"暂不实现"，AI 可能自己加支付、加优惠券、加推荐系统——因为"商城应该有这些"。写了之后，AI 就不会在输出里塞一堆你没要求的东西。

### 技术栈：写到具体框架，不只写"用现代技术"

```text
前端：Vue 3 + Vite + TypeScript + Pinia + Vue Router + Element Plus + Axios
后端：Spring Boot 3 + Java 17 + MyBatis-Plus + MySQL 8 + Redis + RabbitMQ + JWT

架构约束：
- 不使用 JPA。
- 不使用微服务。
- 不使用前端 SSR。
```

技术栈不能只写"用现代技术"。要写到具体框架和版本。

更重要的是写"不使用什么"。不写"不使用 JPA"，AI 可能自己加 JPA。不写"不使用微服务"，AI 可能给你拆出 5 个服务。这些"不使用"和"使用"一样重要。

### 技术要求：每个约束都要具体

```text
- 所有写操作必须说明事务边界。
- 创建订单必须考虑库存不足、重复提交、事务回滚。
- MQ 消息必须说明 exchange、queue、routing key、消息体。
- 消费者必须说明幂等策略。
- Redis 必须说明 key 命名、TTL、缓存未命中和缓存更新。
```

不要只写"考虑事务"和"考虑缓存"。要写清楚"考虑什么"：

- 事务：要说明事务边界，不是只加个 `@Transactional`。
- 缓存：要说明 key 命名、TTL、未命中怎么办、更新策略。
- MQ：要说明 exchange、queue、routing key、消息体、幂等。

这些要求会让 AI 的输出从"提到了事务"变成"说清楚了事务边界在哪、失败怎么回滚"。

### 输出格式：规定交付物清单

```text
1. 产品一句话说明
2. 用户角色
3. 核心业务流程（文本步骤）
4. 功能模块
5. 页面清单
6. 后端接口清单
7. 数据库表设计
8. Redis 缓存设计
9. RabbitMQ 消息设计
10. 事务设计
11. 幂等设计
12. 异常场景
13. 前端目录结构
14. 后端目录结构
15. 开发阶段拆分
16. 每个阶段的验证方式
17. 端到端验收用例
18. 风险点
19. 需要我确认的问题
```

这段的作用是防止 AI 偷懒。不写输出格式，AI 可能只给你一个功能清单和几句描述。写了之后，AI 必须逐项输出，少一项你就能发现。

其中几个特别重要：

- **数据库表设计**：不写这条，AI 可能只写接口不写表结构，到实现时才发现数据模型缺失。
- **事务设计**：不写这条，AI 可能只在代码里加 `@Transactional` 但不说清楚边界。
- **端到端验收用例**：不写这条，AI 输出完就"完成"了，没法验证对不对。
- **需要我确认的问题**：让 AI 把不确定的地方标出来，而不是自己猜。

### 输出约束：禁止空话

```text
- 不要写空泛描述。
- 不要直接写代码。
- 不要引入未声明技术。
- 不要扩大范围。
- 如果缺信息，最多问 5 个问题。
- 如果可以合理假设，请写在"假设"里。
- 每个验收标准必须能被测试。
```

"不要写空泛描述"对应的是 AI 喜欢写的"提升用户体验""保证系统稳定""使用先进架构"。这些话看起来没错，但无法执行、无法验收。

"不要直接写代码"是因为这一轮的目标是产出规范，不是产出实现。代码在后面阶段再写。

"最多问 5 个问题"是防止 AI 无限追问。能合理假设的直接假设，只有影响架构和功能边界的问题才问。

## 验收标准怎么写

"产品完成"不能靠感觉。要让 AI 把验收标准写成可检查项。

差的写法：

```text
系统要稳定可靠，用户体验要好。
```

这种无法测试。

好的写法：

```text
下单验收：
1. 未登录用户调用创建订单接口，返回 401。
2. 商品不存在，返回 404。
3. 库存不足，返回 409，不创建订单。
4. 库存充足，创建订单成功，库存减少。
5. 创建订单失败时，库存不减少。
6. 重复提交同一个幂等键，只生成一个订单。
7. 订单创建后发送 MQ 消息。
8. 订单 30 分钟未支付，自动取消。
9. 已支付订单不会被超时取消。
10. 重复消费超时消息，不会重复取消或重复释放库存。
```

这种可以直接变成测试用例。每一条都有明确的输入、操作和预期结果。

在提示词里可以这样要求：

```text
验收标准要求：

- 每个接口给出请求示例和响应示例。
- 每个核心流程给出正常路径和失败路径。
- 每个页面说明用户能看到什么、能操作什么。
- 每个写操作说明事务失败时的回滚结果。
- 每个 MQ 消费者说明重复消费时不会产生重复业务结果。
- 每个缓存说明命中、未命中、更新、删除的行为。
- 每个权限接口说明未登录、权限不足、正常访问的返回结果。
- 给出最少 10 条端到端验收用例。
```

## 技术方案要拆到什么程度

产品规范不能只有功能清单。要让 AI 把技术细节拆出来。

以"下单"功能为例，不能只写"用户可以下单"。应该让 AI 输出：

```text
下单功能必须说明：

- 前端从购物车或商品详情页发起。
- 后端校验用户登录状态。
- 后端校验商品是否存在、是否上架、库存是否充足。
- 后端在事务内创建订单、订单明细、扣减库存。
- 库存扣减失败时回滚订单。
- 事务提交后发送订单创建 MQ 消息。
- 延迟消息用于订单超时取消。
- 消费者处理超时订单时必须做幂等判断。
- 返回订单 id、订单号、支付截止时间。
```

这才是工程上能落地的描述。每一步都有明确的动作和结果，不是"用户下单然后系统处理"这种废话。

可以在提示词里这样要求：

```text
请为每个核心功能输出：
- 前端页面
- 前端组件
- 前端状态
- 后端接口
- 请求参数
- 响应结构
- 数据库表
- 业务规则
- 事务边界
- 缓存策略
- MQ 消息
- 异常情况
- 测试用例
```

## 让 AI 先问问题

AI 不是每次都应该马上开始设计。如果需求缺关键信息，应该先问。

```text
如果以下信息缺失，请先问我，不要直接假设：
- 目标用户是谁。
- 产品是后台系统、C 端系统还是双端系统。
- 是否需要登录。
- 是否需要支付。
- 是否有管理员。
- 数据是否需要长期保存。
- 是否有并发、库存、金额、权限等高风险逻辑。
```

但也不能让 AI 无限追问。限制：

```text
最多问 5 个问题。
只问影响架构和功能边界的问题。
能合理假设的地方直接假设，并明确写出来。
```

## 让 AI 拆任务

让 AI 直接写完整项目，容易混乱。更好的方式是让它先拆阶段：

```text
请把开发任务拆成可以逐步提交的阶段：

阶段 1：项目初始化
- 前端脚手架
- 后端脚手架
- 数据库连接
- 基础目录结构

阶段 2：用户与鉴权
- 注册
- 登录
- JWT 生成与校验
- 登录拦截器

阶段 3：核心业务
- 商品列表
- 商品详情
- 创建订单
- 库存扣减

阶段 4：异步与可靠性
- RabbitMQ 配置
- 订单创建事件
- 延迟队列
- 死信队列
- 消费幂等

阶段 5：测试与验收
- 单元测试
- 接口测试
- 构建验证
- 关键流程验收
```

任务拆分的目的不是让计划好看，而是让每一步都能独立验证。阶段 1 完成了能跑起来，阶段 2 完成了能登录，阶段 3 完成了能下单——不是等到最后才发现哪一步出了问题。

## 在现有项目里怎么做

如果是在已有项目里做新功能，不要让 AI 重写架构。让它先读项目。

```text
这是一个已有项目。

请先阅读：
- README
- package.json / pom.xml
- 路由配置
- 数据库迁移文件
- controller / service / mapper 目录
- 已有组件和工具函数

然后输出：

1. 当前项目技术栈判断
2. 当前目录结构
3. 这次需求应该改哪些文件
4. 不应该改哪些文件
5. 新增哪些文件
6. 是否需要新增表或字段
7. 是否需要改接口
8. 风险点
9. 实施步骤

约束：
- 不要重写已有架构。
- 不要改无关模块。
- 优先复用已有组件和工具函数。
- 新代码风格要和项目一致。
- 每个阶段完成后都要说明如何验证。
```

这个提示词能减少"顺手重构全项目"的问题。AI 先搞清楚项目现状，再动手改，而不是看到一个 .java 文件就开始重写。

## 二阶段用法

不要一次让 AI 从想法写到代码。拆成三轮。

### 第一轮：产品规范

```text
先输出产品规范和技术方案，不要写代码。
```

目标是让 AI 把需求拆清楚。输出的应该是功能清单、数据模型、接口设计、验收标准——不是代码。

### 第二轮：实施计划

```text
根据刚才的产品规范，输出实施计划。
要求每一步说明：
- 要改哪些文件。
- 要新增哪些文件。
- 为什么这么改。
- 怎么验证。
- 可能影响哪些已有功能。
```

目标是让 AI 把工作拆成可执行步骤。每个步骤有明确的文件、原因和验证方式。

### 第三轮：逐步实现

```text
按实施计划从阶段 1 开始实现。
每完成一个阶段，先停止，输出：
- 已完成内容
- 修改文件
- 验证命令
- 验证结果
- 下一阶段计划
```

目标是避免 AI 一口气改太多。每个阶段完成后停下来验证，确认没问题再继续。

## 让 AI 做自检

提示词最后可以加一段自检要求：

```text
输出后请做一次自检：

- 是否每个功能都有输入、处理、输出。
- 是否每个接口都有参数和响应。
- 是否每张表都有字段、类型和索引。
- 是否每个写操作都有事务边界。
- 是否每个异步消息都有重试、死信和幂等。
- 是否每个缓存都有 key、TTL 和失效策略。
- 是否每个验收标准都能人工或自动测试。
- 是否删除了空泛表述。
```

自检的价值是让 AI 在输出后回头看一遍，发现自己漏了什么。不写这段，AI 输出完就结束了；写了之后，它至少会检查一遍。

## 常见错误

| 错误写法 | 问题 | AI 会怎么反应 |
| --- | --- | --- |
| "做一个完美产品" | 没有范围、没有验收、没有技术边界 | 自己选框架、自己定功能、自己猜深度 |
| "功能尽量完善" | 没有边界 | 擅自扩展需求，加一堆你没要求的东西 |
| "用最好的技术" | 没有指定具体框架 | 乱选框架，可能和团队技术栈不匹配 |
| "直接写代码" | 没有先确认数据模型、接口和业务规则 | 代码能跑但边界缺失，返工成本大 |
| "顺便优化项目" | 没有限制改动范围 | 改无关代码，引入新 bug |
| "你看着办" | 责任边界不清 | 结果完全不可控 |

## 最小提示词

如果不想写长模板，至少写清楚这 8 件事：

```text
1. 我要做什么产品。
2. 给谁用。
3. 必须有哪些功能。
4. 明确不要做哪些功能。
5. 使用什么技术栈（包括不使用什么）。
6. 架构和代码组织有什么要求。
7. 输出什么交付物。
8. 怎么验收。
```

少于这些信息，AI 很容易开始自由创作。

## 去空话检查

- [ ] 是否写清楚产品目标，而不是只写产品名称。
- [ ] 是否写清楚目标用户。
- [ ] 是否区分必须实现和暂不实现。
- [ ] 是否指定前端、后端、数据库、中间件和部署方式。
- [ ] 是否限制 AI 不要引入未声明技术。
- [ ] 是否要求输出接口、表结构、缓存、MQ、事务和异常。
- [ ] 是否要求验收标准可测试。
- [ ] 是否要求 AI 先问关键问题或列出假设。
- [ ] 是否禁止"完美""先进""高可用"这类不可验收空话。
- [ ] 是否把实现拆成阶段，而不是一次性生成全部代码。
