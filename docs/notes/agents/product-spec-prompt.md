# 产品规范提示词

让 AI 做产品，不能只说“帮我做一个完美产品”。

更稳的做法是把提示词写成一份**产品规范 + 技术约束 + 交付协议 + 验收标准**。

AI 真正需要的是这些信息：

- 要解决什么问题。
- 给谁用。
- 哪些功能必须做，哪些暂时不做。
- 用什么技术栈。
- 代码怎么组织。
- 数据怎么存。
- 接口怎么设计。
- 权限、安全、异常、日志、测试怎么处理。
- 怎么判断它做完了。

## 核心原则

好的产品提示词不是“灵感描述”，而是“工程输入”。

```text
模糊输入：
帮我做一个在线商城，要好看，要完善，要能用。

工程输入：
我要做一个面向 C 端用户的秒杀商城 MVP。
前端使用 Vue 3 + Vite + TypeScript + Pinia。
后端使用 Spring Boot 3 + MyBatis-Plus + MySQL + Redis + RabbitMQ。
必须包含登录、商品列表、商品详情、下单、库存扣减、订单超时取消。
先输出 PRD、数据模型、接口设计、任务拆分和验收标准，不要直接写代码。
```

第一种会让 AI 自由发挥。

第二种会让 AI 在你的边界里设计。

## 一份好提示词的结构

```text
角色
  -> 产品背景
  -> 目标用户
  -> 核心目标
  -> 功能范围
  -> 非功能要求
  -> 技术栈
  -> 架构约束
  -> 数据模型
  -> 接口规范
  -> 页面与交互
  -> 权限与安全
  -> 异常与边界场景
  -> 测试要求
  -> 交付物格式
  -> 验收标准
  -> 工作方式
```

不要一次只给“我要做什么”。

要同时告诉 AI “不能怎么做”、“必须交付什么”、“做到什么程度算完成”。

## 可复制模板

下面这个模板适合让 AI 先产出产品规范和技术方案。

不要一上来就让它写代码。

先让它把需求、架构、数据、接口、任务拆清楚。

```text
你是一名资深产品经理、后端架构师、前端架构师和工程负责人。

我现在要做一个产品，请你先不要写代码。
你的任务是根据下面的信息，输出一份可以交给开发执行的产品规范和技术实现方案。

## 1. 产品背景

产品名称：{产品名称}

业务背景：
{为什么要做这个产品，当前有什么问题，用户痛点是什么}

目标用户：
{用户类型，例如普通用户、商家、运营人员、管理员}

核心目标：
{这个产品最重要要解决的 1-3 个目标}

## 2. 功能范围

必须实现：
- {功能 1}
- {功能 2}
- {功能 3}

暂不实现：
- {明确不要做的功能 1}
- {明确不要做的功能 2}

核心业务流程：
1. {用户从哪里进入}
2. {用户做什么操作}
3. {系统怎么响应}
4. {最终产生什么结果}

## 3. 技术栈要求

前端：
- 框架：{Vue 3 / React / Next.js / Nuxt / 其他}
- 语言：{TypeScript / JavaScript}
- 状态管理：{Pinia / Redux / Zustand / 其他}
- UI：{Element Plus / Ant Design / Tailwind CSS / 其他}
- 请求库：{Axios / Fetch / 其他}

后端：
- 框架：{Spring Boot / FastAPI / NestJS / Django / 其他}
- 语言：{Java / Python / TypeScript / 其他}
- ORM：{MyBatis-Plus / JPA / Prisma / SQLAlchemy / 其他}
- 鉴权：{JWT / Session / OAuth2 / 其他}

数据与中间件：
- 数据库：{MySQL / PostgreSQL / MongoDB / 其他}
- 缓存：{Redis / 无}
- 消息队列：{RabbitMQ / Kafka / 无}
- 文件存储：{本地 / S3 / OSS / 无}

部署：
- {Docker / Docker Compose / Kubernetes / 普通服务器}

## 4. 架构约束

请遵守这些约束：
- 不要使用未声明的框架。
- 不要把所有逻辑写在一个文件里。
- 前端按页面、组件、API、类型、状态拆分。
- 后端按 controller、service、mapper/repository、entity、dto、config 拆分。
- 数据库表必须有主键、创建时间、更新时间。
- 写操作必须考虑幂等、事务和异常回滚。
- 涉及缓存时必须说明缓存 key、TTL、失效时机和一致性策略。
- 涉及 MQ 时必须说明 exchange、queue、routing key、消息体、重试、死信和幂等。
- 涉及权限时必须说明哪些接口需要登录，哪些角色能访问。

## 5. 输出要求

请按下面结构输出：

1. 产品一句话说明
2. 用户角色
3. 核心业务流程
4. 功能清单
5. 页面清单
6. 接口清单
7. 数据库表设计
8. 技术架构
9. 前端模块划分
10. 后端模块划分
11. 状态管理设计
12. 缓存设计
13. MQ 设计
14. 权限设计
15. 异常场景
16. 测试用例
17. 开发任务拆分
18. 验收标准
19. 风险点
20. 需要我确认的问题

## 6. 工作方式

- 如果信息不足，先提出最多 5 个关键问题。
- 如果可以合理假设，请直接做出假设，并在“假设”里列出来。
- 不要输出空泛描述，例如“提升用户体验”“保证高可用”。
- 每个技术设计都要说明具体怎么做。
- 每个验收标准都要能被测试。
- 不要直接开始写代码，先输出规范。
```

## 让 AI 指定技术栈

技术栈不能只写“用现代技术”。

应该写到具体框架、版本边界和不允许使用的东西。

```text
技术栈要求：

- 前端必须使用 Vue 3 + Vite + TypeScript。
- 路由使用 Vue Router。
- 状态管理使用 Pinia。
- HTTP 请求统一封装在 src/api。
- UI 使用 Element Plus，不要混用其他 UI 库。
- 后端使用 Spring Boot 3。
- 数据访问使用 MyBatis-Plus，不使用 JPA。
- 数据库使用 MySQL 8。
- 缓存使用 Redis。
- 消息队列使用 RabbitMQ。
- 接口风格使用 RESTful。
- 鉴权使用 JWT。
- 不使用微服务，先做单体分层架构。
```

这样 AI 不容易乱引入框架。

如果你不写“不使用 JPA”“不使用微服务”，AI 可能自己加。

## 让 AI 输出技术方案

产品规范不能只有功能清单。

要让 AI 把技术细节拆出来。

```text
请为每个核心功能输出：

- 前端页面
- 前端组件
- 前端状态
- 后端接口
- 请求参数
- 响应结构
- 数据库表
- 业务规则
- 事务边界
- 缓存策略
- MQ 消息
- 异常情况
- 测试用例
```

例如“下单”功能，不能只写“用户可以下单”。

应该让 AI 输出：

```text
下单功能必须说明：

- 前端从购物车或商品详情页发起。
- 后端校验用户登录状态。
- 后端校验商品是否存在、是否上架、库存是否充足。
- 后端在事务内创建订单、订单明细、扣减库存。
- 库存扣减失败时回滚订单。
- 事务提交后发送订单创建 MQ 消息。
- 延迟消息用于订单超时取消。
- 消费者处理超时订单时必须做幂等判断。
- 返回订单 id、订单号、支付截止时间。
```

这才是工程上能落地的描述。

## 让 AI 先问问题

AI 不是每次都应该马上开始设计。

如果需求缺关键信息，应该先问。

```text
如果以下信息缺失，请先问我，不要直接假设：

- 目标用户是谁。
- 产品是后台系统、C 端系统还是双端系统。
- 是否需要登录。
- 是否需要支付。
- 是否有管理员。
- 数据是否需要长期保存。
- 是否有并发、库存、金额、权限等高风险逻辑。
```

但是不能让 AI 无限追问。

推荐限制：

```text
最多问 5 个问题。
只问影响架构和功能边界的问题。
能合理假设的地方直接假设，并明确写出来。
```

## 让 AI 拆任务

让 AI 直接写完整项目，容易混乱。

更好的方式是让它先拆任务。

```text
请把开发任务拆成可以逐步提交的阶段：

阶段 1：项目初始化
- 前端脚手架
- 后端脚手架
- 数据库连接
- 基础目录结构

阶段 2：用户与鉴权
- 注册
- 登录
- JWT 生成与校验
- 登录拦截器

阶段 3：核心业务
- 商品列表
- 商品详情
- 创建订单
- 库存扣减

阶段 4：异步与可靠性
- RabbitMQ 配置
- 订单创建事件
- 延迟队列
- 死信队列
- 消费幂等

阶段 5：测试与验收
- 单元测试
- 接口测试
- 构建验证
- 关键流程验收
```

任务拆分的目的不是让计划好看。

而是让每一步都能独立验证。

## 让 AI 写验收标准

“产品完成”不能靠感觉。

要把验收标准写成可检查项。

```text
验收标准要求：

- 每个接口给出请求示例和响应示例。
- 每个核心流程给出正常路径和失败路径。
- 每个页面说明用户能看到什么、能操作什么。
- 每个写操作说明事务失败时的回滚结果。
- 每个 MQ 消费者说明重复消费时不会产生重复业务结果。
- 每个缓存说明命中、未命中、更新、删除的行为。
- 每个权限接口说明未登录、权限不足、正常访问的返回结果。
- 给出最少 10 条端到端验收用例。
```

示例：

```text
下单验收：

1. 未登录用户调用创建订单接口，返回 401。
2. 商品不存在，返回 404。
3. 库存不足，返回 409，不创建订单。
4. 库存充足，创建订单成功，库存减少。
5. 创建订单失败时，库存不减少。
6. 重复提交同一个幂等键，只生成一个订单。
7. 订单创建后发送 MQ 消息。
8. 订单 30 分钟未支付，自动取消。
9. 已支付订单不会被超时取消。
10. 重复消费超时消息，不会重复取消或重复释放库存。
```

这种验收标准可以直接变成测试用例。

## 让 AI 避免空话

提示词里要明确禁止空话。

```text
禁止输出：

- “提升用户体验”
- “保证系统稳定”
- “使用先进架构”
- “代码要优雅”
- “后续可扩展”

除非你能写清楚：

- 具体怎么提升。
- 具体怎么保证。
- 使用什么架构。
- 哪个模块怎么拆。
- 哪个扩展点在哪里。
- 用什么测试证明。
```

可以让 AI 做自检：

```text
输出后请做一次自检：

- 是否每个功能都有输入、处理、输出。
- 是否每个接口都有参数和响应。
- 是否每张表都有字段、类型和索引。
- 是否每个写操作都有事务边界。
- 是否每个异步消息都有重试、死信和幂等。
- 是否每个缓存都有 key、TTL 和失效策略。
- 是否每个验收标准都能人工或自动测试。
- 是否删除了空泛表述。
```

## 让 AI 处理现有项目

如果是在已有项目里做产品，不要让 AI 重写架构。

应该让它先读项目。

```text
这是一个已有项目。

请先阅读：

- README
- package.json / pom.xml
- 路由配置
- 数据库迁移文件
- controller / service / mapper 目录
- 已有组件和工具函数

然后输出：

1. 当前项目技术栈判断
2. 当前目录结构
3. 这次需求应该改哪些文件
4. 不应该改哪些文件
5. 新增哪些文件
6. 是否需要新增表或字段
7. 是否需要改接口
8. 风险点
9. 实施步骤

约束：

- 不要重写已有架构。
- 不要改无关模块。
- 优先复用已有组件和工具函数。
- 新代码风格要和项目一致。
- 每个阶段完成后都要说明如何验证。
```

这个提示词适合给代码 Agent。

它能减少“顺手重构全项目”的问题。

## 完整示例

下面是一个更接近实战的完整提示词。

```text
你是一名资深产品经理、Java 后端架构师、Vue 前端架构师和工程负责人。

我要做一个秒杀商城 MVP。
请先不要写代码，先输出产品规范和技术实现方案。

## 产品背景

我要做一个用于学习 Java 后端工程能力的秒杀商城。
它不是商业级大平台，但要覆盖真实后端项目里的核心工程问题：
登录、商品、库存、订单、缓存、MQ、事务、幂等、异常和接口设计。

## 目标用户

- 普通用户：浏览商品、查看详情、下单。
- 管理员：维护商品和库存。

## 必须实现

- 用户注册和登录。
- JWT 鉴权。
- 商品列表。
- 商品详情。
- 创建订单。
- 库存扣减。
- 订单超时取消。
- RabbitMQ 发送订单创建事件。
- RabbitMQ 延迟消息处理超时订单。
- Redis 缓存商品详情。
- MySQL 保存用户、商品、库存、订单。

## 暂不实现

- 在线支付。
- 优惠券。
- 物流。
- 多商户。
- 微服务拆分。
- 分布式事务。

## 技术栈

前端：
- Vue 3
- Vite
- TypeScript
- Pinia
- Vue Router
- Element Plus
- Axios

后端：
- Spring Boot 3
- Java 17
- MyBatis-Plus
- MySQL 8
- Redis
- RabbitMQ
- JWT

架构：
- 单体分层架构。
- 不使用 JPA。
- 不使用微服务。
- 不使用前端 SSR。

## 技术要求

- 后端按 controller、service、mapper、entity、dto、config、common 分层。
- 前端按 views、components、api、stores、router、types 分层。
- 所有写操作必须说明事务边界。
- 创建订单必须考虑库存不足、重复提交、事务回滚。
- MQ 消息必须说明 exchange、queue、routing key、消息体。
- 消费者必须说明幂等策略。
- Redis 必须说明 key 命名、TTL、缓存未命中和缓存更新。
- 接口必须给出请求参数和响应格式。
- 数据库表必须给出字段、类型、索引和说明。

## 输出格式

请输出：

1. 产品一句话说明
2. 用户角色
3. 核心业务流程图，用文本步骤表示
4. 功能模块
5. 页面清单
6. 后端接口清单
7. 数据库表设计
8. Redis 缓存设计
9. RabbitMQ 消息设计
10. 事务设计
11. 幂等设计
12. 异常场景
13. 前端目录结构
14. 后端目录结构
15. 开发阶段拆分
16. 每个阶段的验证方式
17. 端到端验收用例
18. 风险点
19. 需要我确认的问题

## 输出约束

- 不要写空泛描述。
- 不要直接写代码。
- 不要引入未声明技术。
- 不要扩大范围。
- 如果缺信息，最多问 5 个问题。
- 如果可以合理假设，请写在“假设”里。
- 每个验收标准必须能被测试。
```

这类提示词的价值是：AI 输出的不是一段“项目介绍”，而是一份可以继续推进开发的工程规范。

## 二阶段用法

不要一次让 AI 从想法写到代码。

推荐拆成三轮。

### 第一轮：产品规范

```text
先输出产品规范和技术方案，不要写代码。
```

目标是让 AI 把需求拆清楚。

### 第二轮：实施计划

```text
根据刚才的产品规范，输出实施计划。
要求每一步说明：

- 要改哪些文件。
- 要新增哪些文件。
- 为什么这么改。
- 怎么验证。
- 可能影响哪些已有功能。
```

目标是让 AI 把工作拆成可执行步骤。

### 第三轮：逐步实现

```text
按实施计划从阶段 1 开始实现。
每完成一个阶段，先停止，输出：

- 已完成内容
- 修改文件
- 验证命令
- 验证结果
- 下一阶段计划
```

目标是避免 AI 一口气改太多。

## 常见错误

| 错误写法 | 问题 |
| --- | --- |
| “做一个完美产品” | 没有范围、没有验收、没有技术边界 |
| “功能尽量完善” | AI 会擅自扩展需求 |
| “用最好的技术” | AI 会乱选框架 |
| “直接写代码” | 没有先确认数据模型、接口和业务规则 |
| “顺便优化项目” | 容易改无关代码 |
| “你看着办” | 责任边界不清，结果不可控 |

## 最小提示词

如果你不想写长模板，至少写清楚这 8 件事：

```text
1. 我要做什么产品。
2. 给谁用。
3. 必须有哪些功能。
4. 明确不要做哪些功能。
5. 使用什么技术栈。
6. 架构和代码组织有什么要求。
7. 输出什么交付物。
8. 怎么验收。
```

少于这些信息，AI 很容易开始“自由创作”。

## 去空话检查

- [ ] 是否写清楚产品目标，而不是只写产品名称。
- [ ] 是否写清楚目标用户。
- [ ] 是否区分必须实现和暂不实现。
- [ ] 是否指定前端、后端、数据库、中间件和部署方式。
- [ ] 是否限制 AI 不要引入未声明技术。
- [ ] 是否要求输出接口、表结构、缓存、MQ、事务和异常。
